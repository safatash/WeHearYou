Three ways to build Phase 1

Same four features, meaningfully different data architectures. The choice affects how fast it is to build, how much you can scale, and what Phase 2–5 looks like.

Option A — Minimal footprint (fastest to ship)
💬 Reply publishing
Add replyPublishedAt + replyGbpId to existing Review model. "Publish" button calls GBP API.
📝 GBP Posts
New GbpPost table with draft/scheduled/published status. Cron job publishes scheduled items.
🖼 Photos
Stateless API bridge only. Upload → GBP API. View live photos fetched on demand. No local storage.
❓ Q&A
Stateless API bridge. Fetch questions live, submit answers via API. No local storage.
Auto-publish: New PUBLISH_GBP_REPLY automation step type — plugs into existing automation engine.

Pros: Ships fastest (~4–5 weeks). Minimal schema changes. Cons: Photos and Q&A have no history. UX is inconsistent — some data is live, some is local. Hard to add analytics later.

Option B — Unified GBP content model ★ Recommended
All four content types get a local record. Everything follows the same lifecycle: draft → scheduled → published → synced.
New database tables
GbpPost
type · content · scheduledAt · publishedAt · gbpPostId
GbpPhoto
url · category · scheduledAt · publishedAt · gbpMediaId
GbpQuestion
questionText · answerText · answeredAt · gbpQuestionId
Review (extended)
+ replyPublishedAt · replyGbpId (already has replyDraft)
One cron job handles all scheduled items across all types. One PUBLISH_GBP_REPLY automation step. Consistent history & audit trail across the whole GBP hub.

Pros: Consistent UX, full history, enables analytics in Phase 2+, clean foundation for rank tracking. Cons: ~1–2 weeks more upfront. Must handle "items published directly on Google" with periodic re-sync.

Option C — Action queue (most scalable)
Every GBP write becomes an event in a GbpActionQueue table. A single worker processes all of them.
GbpActionQueue {
  type: PUBLISH_REPLY | CREATE_POST | UPLOAD_PHOTO | ANSWER_QA
  payload: JSON
  status: pending | processing | done | failed
  scheduledAt: DateTime?
  retryCount: Int
}
Works well at scale — but for Phase 1, querying "what posts are scheduled?" means joining through the queue payload, which is awkward. Better suited once you have 1000s of daily actions.

Pros: Unified audit log, retryable, handles failures gracefully. Cons: Over-engineered for Phase 1. Harder to query current state for the UI. We'd likely refactor toward B anyway.
