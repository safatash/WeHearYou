Three approaches to the SEO architecture

The main decision is where SEO logic lives — this affects how maintainable it is as more pages are added

A

Extend in place (minimal change)

Add generateMetadata directly to /b/[slug]/page.tsx, extend buildLocalBusinessSchema in public-profile.ts, add app/sitemap.ts as a standalone file.

Pros

  • Least code change
  • Follows existing pattern

Cons

  • SEO logic scattered across files
  • Harder to reuse for other pages later
B

Dedicated SEO lib ★ Recommended

Create src/lib/seo.ts with buildLocationMetadata() and the enhanced buildLocalBusinessSchema(). The page and sitemap both import from this one place.

Pros

  • All SEO logic in one file
  • Easy to extend to other pages
  • Clean page file

Cons

  • New file to maintain
C

next-sitemap package

Use the next-sitemap npm package for automatic sitemap + robots.txt generation at build time. Add generateMetadata and schema enhancements separately.

Pros

  • Handles robots.txt automatically
  • Battle-tested package

Cons

  • Adds a dependency
  • Static — misses newly created locations until next build
  • Overkill for what we need
